Document: DFlash Dataset Generation Plan for GLM-5.1 Speculative Decoding Draft Model1. Problem DescriptionWe need to generate a high-quality training dataset for DFlash (block-diffusion speculative decoding) to create a small ~0.8B draft model that can be used with the 754B-parameter GLM-5.1 MoE teacher (Zhipu AI).Key requirements for the DFlash dataset (per the 2026 paper):50K–800K sequences (start with 50K–100K for PoC, scale to 200K–400K for good quality).
Each sequence: packed short examples (initially 64–128 tokens) into longer contexts (target 4K–16K token windows).
For every sequence we must extract:Generated tokens from the teacher.
Hidden-layer context vectors from exactly 5 uniformly sampled layers (between layer 2 and third-to-last).
512 random anchor positions per sequence for block-diffusion training.
Top-K or full next-token distributions (optional but useful).

Core compute challenge:Full GLM-5.1 model (quantized with Unsloth UD-Q2_K_XL) is 205 GB total:15 GB dense layers.
190 GB MoE experts.

Even on a high-memory Colab TPUv6 instance (~173 GB system RAM / 225 GB disk), the full model cannot be loaded at once for efficient inference.
Naïve full-model loading + batched prefill is SSD-bound and extremely slow for the required dataset size.
We must support layer-wise expert deduplication (only load active experts per token) and selective hidden-state extraction to avoid materializing massive activations/KV caches.

Without a smart sharding + streaming strategy, dataset generation becomes impractical on consumer/cloud TPUs and would require a full cluster.2. Proposed Solution / Way We Want to Address ItWe will implement layer-wise, pipeline-parallel inference on TPUs using the Unsloth UD-Q2_K_XL weights. The model will be split into two halves so that each half (approximately half the layers) fits comfortably inside a single 32 GB TPU slice.High-Level WorkflowOffline dataset preparation (this phase):Start with curated short sequences (64–128 tokens) from high-quality sources (e.g., Nemotron Post-Training Dataset V2 + code/data mixtures).
Pack them into 4K (or up to 16K) windows depending on what the TPU can handle efficiently.
Run teacher inference layer-by-layer:Load weights from RAM/disk for only the active experts (MoE expert deduplication).
Prefill one layer at a time.
When a target hidden layer (one of the 5 sampled layers) is reached → save the fused compact context vector.
Discard intermediate activations immediately.
Move to the next layer.

Continue until System RAM is exhausted → repeat with the next batch of packed sequences.
Model splitting: Split GLM-5.1 into two chunks (layers 0–N and N+1–end). Run first half on TPU #1, second half on TPU #2 (pipeline parallelism across the two TPUs).
Output: Serialized DFlash-ready dataset shards stored on Google Cloud Storage (GCS) containing tokens + 5× fused hidden context vectors + anchor masks.

Later phase (draft training):Stream the precomputed dataset from GCS → Colab TPUv6 high-mem instance.
Train the 0.8B draft model using the standard DFlash training recipe (no teacher needed at this stage).

Why This Approach WorksMemory footprint: Splitting into two halves + loading only active MoE experts + processing one layer at a time keeps each TPU well under 32 GB.
Compute efficiency: Packing + layer-wise streaming maximizes TPU utilization and avoids SSD bottlenecks.
Scalability: Can run on 1–2 Colab TPUv6 high-mem instances (or more if we scale to multiple pipelines). No full cluster required for the training-data-generation step.
DFlash compatibility: Hidden extraction happens exactly where the paper expects it; the draft model never sees the full teacher.

3. Deliverables Expected from the Next AgentPlease create a detailed implementation plan covering:Exact model splitting strategy (which layers go to TPU #1 vs #2).
JAX / PyTorch-XLA code skeleton for layer-wise prefill + expert deduplication + hidden extraction.
Data packing logic (64/128 → 4K/16K) and GCS streaming.
Memory profiling & checkpointing strategy (what to save when RAM is exhausted).
Estimated runtime/storage for 50K, 100K, and 200K sequences.
Any required modifications to Unsloth UD-Q2_K_XL loading for TPU.
Testing plan (small-scale smoke test on 1K sequences).
